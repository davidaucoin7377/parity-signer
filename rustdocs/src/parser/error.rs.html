115/// Errors directly related to transaction parsing
116///
117/// Signable transactions are differentiated based on prelude:
118///
119/// - `53xx00` mortal transactions
120/// - `53xx02` immortal transactions
121/// - `53xx03` text message transactions
122///
123/// `53xx00` and `53xx02` transactions contain encoded transaction data, and
124/// are parsed prior to signing using the network metadata. Transaction is
125/// generated in client, for certain address and within certain network.
126/// To parse the transaction and to generate the signature, Vault must
127/// have the network information (network specs and correct network metadata)
128/// and the public address-associated information in its database.
129///
130/// `53xx00` and `53xx02` transcations consist of:
131///
132/// - prelude, `53xx00` or `53xx02`, where `xx` stands for the encryption
133///   algorithm associated with address and network used
134/// - public key corresponding to the address that can sign the transaction
135/// - encoded call data, the body of the transaction
136/// - extensions, as set in the network metadata
137/// - genesis hash of the network in which the transaction was generated
138///
139/// Parsing process first separates the prelude, public key, genesis hash and
140/// the combined call + extensions data.
141///
142/// The call information is SCALE-encoded into `Vec<u8>` bytes and then those
143/// bytes are SCALE-encoded again, so that the call data contained in the
144/// transaction consists of `compact` with encoded call length in bytes
145/// followed by the `Vec<u8>` with the encoded data.
146///
147/// Call and extensions are cut based on the call length declared at the start
148/// of the combined call + extensions data.
149///
150/// Then the extensions are decoded, and it is checked that the metadata version
151/// in extensions coincides with the metadata version used for the decoding.
152///
153/// Decoding the extensions for metadata with `RuntimeMetadataV12` or
154/// `RuntimeMetadataV13` is using a static set of extensions, namely:
155///
156/// - [`Era`](https://docs.rs/sp-runtime/6.0.0/sp_runtime/generic/enum.Era.html)
157/// - nonce, compact `u64`
158/// - transaction tip, compact `u128`
159/// - metadata version, `u32`
160/// - tx version, `u32`
161/// - network genesis hash, `[u8; 32]`
162/// - block hash, `[u8; 32]`
163///
164/// Decoding the extensions for metadata with `RuntimeMetadataV14` uses
165/// dynamically acquired set of extensions from the metadata itself.
166///
167/// After the extensions, the call data itself is decoded using the network
168/// metadata. Each call first byte is the index of the pallet.
169///
170/// Metadata with `RuntimeMetadataV12` or `RuntimeMetadataV13` has only type
171/// names associated with call arguments. Vault finds what the types really
172/// are and how to decode them by using the types information that must be in
173/// Vault database.
174/// For `RuntimeMetadataV12` or `RuntimeMetadataV13` the second byte in call is
175/// the index of the method within the pallet, and thes Vault finds the types
176/// used by the method and proceeds to decode the call data piece by piece.
177///
178/// Metadata with `RuntimeMetadataV14` has types data in-built in the metadata
179/// itself, and the types needed to decode the call are resolved during the
180/// decoding. For `RuntimeMetadataV14` the second byte in call is also
181/// the index of the method within the pallet, but this already goes into the
182/// type resolver.
183///
184/// Calls may contain nested calls, for `RuntimeMetadataV12` or
185/// `RuntimeMetadataV13` metadata the call decoding always starts with pallet
186/// and method combination processing. For `RuntimeMetadataV14` metadata the
187/// nested calls are processed through the type resolver, i.e. the pallet index
188/// is processed independently only on the start of the decoding.
189///
190/// `53xx03` transaction consists of:
191///
192/// - prelude `53xx03`, where `xx` stands for the encryption algorithm
193///   associated with address and network used
194/// - public key corresponding to the address that can sign the transaction
195/// - SCALE-encoded `String` contents of the message
196/// - genesis hash of the network in which the transaction was generated
197///
198/// Vault assumes that every byte of the transaction will be processed, and
199/// shows an error if this is not the case.
