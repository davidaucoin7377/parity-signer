1//! This crate is intended to support the
2//! [Vault](https://github.com/paritytech/parity-signer) from the active
3//! (non air-gapped) side.
4//!
5//! This crate is mainly used to:
6//!
7//! - fetch network data through RPC calls
8//! - prepare Vault update and derivation import payloads
9//! - generate Vault update QR codes, either signed or unsigned, and
10//!   derivations import QR codes, to be scanned into Vault
11//! - maintain the `hot` database on the network-connected device, to store and
12//!   manage the data that went into update QR codes
13//! - maintain Vault default network metadata set in `defaults` crate and
14//!   prepare the `cold` database for the Vault release
15//!
16//! # Supported Vault updates
17//!
18//! Crate `generate_message` can generate and the Vault can accept following
19//! updates:
20//!
21//! - `add-specs`, to add a new network (i.e. the network specs) into the Vault
22//! - `load-metadata`, to load into the Vault the network metadata, for
23//!   networks that already have corresponding network specs entry in the Vault
24//!   database
25//! - `load-types`, to load types information (it is used to support the
26//!   transactions parsing in networks with legacy metadata, `RuntimeMetadata`
27//!   version below `V14`)
28//!
29//! Updates are assembled as `Vec<u8>` and could be transformed into:
30//!
31//! - `PNG` QR codes, static or dynamic multiframe depending on the data size
32//! - hex-encoded string (for tests)
33//!
34//! Information in `add-specs`, `load-metadata` and `load-types` could be either
35//! signed or unsigned. Using signed updates is strongly encouraged.
36//!
37//! Update has following general structure:
38//!
39//! <table>
40//!     <tr>
41//!         <td>prelude <code>[0x53, 0x<encryption code>, 0x<payload code>]</code></td>
42//!         <td>verifier public key (if signed)</td>
43//!         <td>update payload</td>
44//!         <td>signature (if signed)</td>
45//!         <td>reserved tail, currently empty</td>
46//!     </tr>
47//! </table>
48//!
49//! `<encryption code>` indicates encryption algorithm that was used
50//! to sign the update:
51//!
52//! <table>
53//!     <tr>
54//!         <td><code>0x00</code></td>
55//!         <td>Ed25519</td>
56//!     </tr>
57//!     <tr>
58//!         <td><code>0x01</code></td>
59//!         <td>Sr25519</td>
60//!     </tr>
61//!     <tr>
62//!         <td><code>0x02</code></td>
63//!         <td>Ecdsa</td>
64//!     </tr>
65//!     <tr>
66//!         <td><code>0xff</code></td>
67//!         <td>unsigned</td>
68//!     </tr>
69//! </table>
70//!
71//! Update payloads content is described in [definitions::qr_transfers].
72//!
73//! <table>
74//!     <tr>
75//!         <th>update payload</th>
76//!         <th>update content type</th>
77//!         <th>data signed, <code>to_sign</code> form</th>
78//!         <th>data in payload, <code>to_transfer</code> form</th>
79//!     </tr>
80//!     <tr>
81//!         <td><code>add_specs</code></td>
82//!         <td><code>ContentAddSpecs</code></td>
83//!         <td>SCALE encoded <code>NetworkSpecs</code></td>
84//!         <td>double SCALE encoded <code>NetworkSpecs</code></td>
85//!     </tr>
86//!     <tr>
87//!         <td><code>load_metadata</code></td>
88//!         <td><code>ContentLoadMeta</code></td>
89//!         <td>concatenated SCALE encoded metadata vector and network genesis hash</td>
90//!         <td>concatenated SCALE encoded metadata vector and network genesis hash</td>
91//!     </tr>
92//!     <tr>
93//!         <td><code>load_types</code></td>
94//!         <td><code>ContentLoadTypes</code></td>
95//!         <td>SCALE encoded <code>Vec&ltTypeEntry&gt</code></td>
96//!         <td>double SCALE encoded <code>Vec&ltTypeEntry&gt</code></td>
97//!     </tr>
98//! </table>
99//!
100//! Note that the update payloads are build in such a way that the length of
101//! the payload always could be easily found, thus allowing to separate update
102//! payload, signature and reserved tail in Vault when accepting the update.
103//! The tail is reserved to future-proof the updates if the multi-signing is
104//! ever implemented for them. Currently the tail is empty.
105//!
106//! # Updates generation
107//!
108//! Updates are generated in following stages:
109//!
110//! 1. make update payload
111//! 2. (optional) make signature for update payload
112//! 3. make update QR code (optionally signed), that could be scanned into
113//!    Vault
114//!
115//! Steps (1) and (3) are done in `generate_message`, the signature is produced
116//! in other tools, except the test "signed" updates with Alice as a verifier,
117//! when the signature is produced while making QR code during step (3).
118//!
119//! Signature could be produced with Subkey or with Vault. For update signing
120//! it is recommended to use a dedicated key, not used for transactions. This
121//! way, if the signed data was not really the update data, but something else
122//! posing as the update data, the signature produced could not do any damage.
123//!
124//! If the Vault is used to produce the signature, it should be a dedicated
125//! Vault with no verifier or weak key verifier for the network: before the
126//! signature is produced, an unsigned or easily signed update must be loaded
127//! into Vault.
128//!
129//! # Available commands
130//!
131//! ## Display content of the metadata `METATREE` tree of the hot database
132//!
133//! `$ cargo run show metadata`
134//!
135//! Prints for each entry in hot database [`METATREE`](constants::METATREE)
136//! tree:
137//!
138//! - network name
139//! - network metadata version
140//! - hexadecimal metadata hash
141//! - hexadecimal block hash for the block at which the metadata was fetched
142//!
143//! Note that for each network a maximum of 2 metadata entries is stored in the
144//! hot database at any time.
145//!
146//! ## Display content of the address book `ADDRESS_BOOK` tree of the hot database
147//!
148//! `$ cargo run show networks`
149//!
150//! Prints for each entry in hot database
151//! [`ADDRESS_BOOK`](constants::ADDRESS_BOOK) tree:
152//!
153//! - address book title for the network `<network_name>-<network_encryption>`,
154//!   used only to distinguish between address book entries
155//! - URL address at which RPC calls are made for the network
156//! - network encryption
157//! - additional marker that the network is a default one, i.e. entry has not
158//!   changed since the database generation
159//! - network title as it will be displayed in Vault, from
160//!   [`NetworkSpecs`](definitions::network_specs::NetworkSpecs)
161//!
162//! ## Show network specs for a network, as recorded in the hot database
163//!
164//! `$ cargo run show specs <ADDRESS BOOK TITLE>`
165//!
166//! Prints network address book title and corresponding
167//! [`NetworkSpecs`](definitions::network_specs::NetworkSpecs)
168//! from [`SPECSTREEPREP`](constants::SPECSTREEPREP) tree of the hot
169//! database.
170//!
171//! ### Example
172//!
173//! `$ cargo run show specs westend-sr25519`
174//!
175//! ## Check external file with hex-encoded metadata
176//!
177//! `$ cargo run show check-file <METADATA FILE>`
178//!
179//! Asserts that:
180//!
181//! - the file contains valid metadata, with retrievable network name and
182//!   version
183//! - if the metadata for same network name and version is in the hot
184//!   database, it completely matches the one from the file
185//!
186//! ### Example
187//!
188//! `$ cargo run show check-file "../defaults/release_metadata/kusama9230"`
189//!
190//! ## Show metadata fetch block history from `META_HISTORY` tree of the hot database
191//!
192//! `$ cargo run show block-history`
193//!
194//! Prints block hashes at which the network metadata was fetched as it first
195//! got in the database. If the metadata is from `.wasm` file, there is no entry
196//! until a proper metadata fetch from a node is done with some associated block
197//! hash.
198//!
199//! [`META_HISTORY`](constants::META_HISTORY) tree stores all block hashes that
200//! were ever encountered on successful new metadata fetch, and clears only on
201//! the database reset.
202//!
203//! Block hashes could be useful should silent metadata updates (metadata change
204//! with no version bump) happen again.

206//! ## Prepare `add_specs` update payload
207//!
208//! `$ cargo run add-specs [OPTIONS] <-d|-f|-k|-p|-t> <--all|--name <NAME>|--url <ADDRESS>>`

223//!   calls, produce output files

258//! Reference keys (exactly only one has to be used):
259//!
260//! - `--all`: all networks with entries in the
261//!   [`ADDRESS_BOOK`](constants::ADDRESS_BOOK) tree of the hot database
262//! - `--name` followed by single network address book title: for a network with
263//!   existing record in the [`ADDRESS_BOOK`](constants::ADDRESS_BOOK)
264//! - `--url` followed by single URL address: reserved for networks with no
265//!   record yet in the [`ADDRESS_BOOK`](constants::ADDRESS_BOOK)
266//!
267//! `--all` key could be used with `--pass-errors` key, to stop processing after first
268//! error.
269//!
270//! `--encryption` key to override specifying encryption algorithm supported by the
271//! network is optional for `--name` reference key (since there is already an entry in
272//! the database with specified encryption) and mandatory for `--url` reference key.
273//! Supported variants are:
274//!
275//! - `ed25519`
276//! - `sr25519`
277//! - `ecdsa`
278//!
279//! Sequence invoking token override could be used when processing an
280//! individual network that has multiple allowed decimals and unit values
281//! retrieved as arrays of equal size. To override token, key `--token-decimals`
282//! followed by `u8` decimals value and key `--token-unit` `String` unit value is used.
283//! By default, if no token override in provided, such networks have `0u8` decimals
284//! and `UNIT` unit set up.
285//!
286//! Title override could be used when processing an individual network, to set
287//! the title under which the network will be displayed in Vault, should the
288//! `add-specs` payload be accepted. Non-default networks, if the title override
289//! is not specified, have title `<network_name>-<network_encryption>`.
290//!
291//! Not all setting and reference key combinations are compatible, and not all
292//! overrides are supported. Users are encouraged to comment if they need some
293//! other than current key combinations available.

621//! `$ cargo run load-metadata -a`
622//!
623//! Make `load_metadata` update payload for a network not in the database:
624//!
625//! `$ cargo run load-metadata -d -u wss://rococo-rpc.polkadot.io`
626//!
627//! ## Prepare `load_types` update payload
628//!
629//! `$ cargo run load-types`
630//!
631//! A file is generated in dedicated [`FOLDER`](constants::FOLDER) to
632//! (optionally) be signed and later be transformed into `load_types` update QR.
633//! Output file name is `sign_me_load_types`.
634//!
635//! ## Generate update QR and/or hexadecimal string file
636//!
637//! Raw `[u8]` update payloads, as prepared by `add_specs`, `load_metadata` or
638//! `load_types` commands get transformed into update QR codes (to be scanned
639//! into the Vault) or textfiles with hexadecimal data (for tests).
640//!
641//! There are two commands for generating updates: `make` and `sign`.
642//!
643//! Command `make` is used to generate:
644//!
645//! - signed updates with a valid signature, associated public key and
646//!   encryption algorithm
647//! - test signed updates, i.e. updates signed by a key with
648//!   [Alice seed phrase](constants::ALICE_SEED_PHRASE) and derivation `//Alice`,
649//!   with encryption algorithm chosen by user, for tests
650//! - unsigned updates
651//!
652//! Signature for `make` command is generated for contents of raw `[u8]` update
653//! payload file using, for example, Subkey.
654//!
655//! Command `sign` is used to generate signed updates with a valid
656//! [`SufficientCrypto`](definitions::crypto::SufficientCrypto) produced by
657//! Vault. Vault exports `SufficientCrypto` produced for one of its keys
658//! as a static QR code, this QR code content goes into command line.
659//!
660//! Update QR and/or hexadecimal string file are produced in
661//! [`EXPORT_FOLDER`](constants::EXPORT_FOLDER).
662//!
663//! Keys and most arguments (except file paths) are not case-sensitive.
664//!
665//! The validity of the signature or `SufficientCrypto`, if provided, is checked
666//! before assembling update.
667//!
668//! The payload is checked to be valid, with decodeable content. If default
669//! output file name is used, it is generated based on the payload content.
670//!
671//! <table>
672//!     <tr>
673//!         <th><code>msg</code></th>
674//!         <th>default update file name</th>
675//!     </tr>
676//!     <tr>
677//!         <td><code>add-specs</code></td>
678//!         <td><code>add_specs_&ltnetwork_name&gt-&ltnetwork_encryption&gt</code></td>
679//!     </tr>
680//!     <tr>
681//!         <td><code>load-metadata</code></td>
682//!         <td><code>load_metadata_&ltnetwork_name&gtV&ltmetadata_version&gt</code></td>
683//!     </tr>
684//!     <tr>
685//!         <td><code>load-types</code></td>
686//!         <td><code>load_types</code></td>
687//!     </tr>
688//! </table>
689//!
690//! Names for Alice-signed updates have additional tail
691//! `_Alice-<alice_signature_encryption>`.
692//!
693//! Names for unsigned updates have additional tail `_unsigned`.
694//!
695//! ### `make` command
696//!
697//! `$ cargo run make <keys> <arguments>`
698//!
699//! Keys to be used in command line:
700//!
701//! - Key `--goal` followed by the type to generate
702//!    - `qr` will generate only a png QR code
703//!    - `text` will generate only text file with hex-encoded update.
704//!    - default, i.e. if goal is not provided, both QR code and text file are generated.
705//!
706//! - Key `--crypto` followed by encryption used to make update signature:
707//!    - `ed25519`
708//!    - `sr25519`
709//!    - `ecdsa`
710//!    - `none` if the message is not verified
711//!
712//! - Key `--msg` followed by update type:
713//!    - `load-types`
714//!    - `load-metadata`
715//!    - `add-specs`
716//!
717//! - Key `--verifier` (can be entered only if the `--crypto` argument was
718//!   `ed25519`, `sr25519`, or `ecdsa`), followed by:
719//!    - `Alice` to generate messages "verified" by
720//!       [Alice seed phrase](constants::ALICE_SEED_PHRASE) with derivation `//Alice`
721//!    - `-hex` followed by hex public key
722//!    - `-file` followed by the path in dedicated [`FOLDER`](constants::FOLDER)
723//!       for file with public key as raw bytes
724//!
725//! - Key `--payload` followed by file path in dedicated
726//!   [`FOLDER`](constants::FOLDER) containing already generated payload as
727//!   raw bytes
728//!
729//! - Key `--signature` (can be entered only if the `--crypto` argument was
730//!   `ed25519`, `sr25519`, or `ecdsa` **and** `--verifier` is not `Alice`),
731//!   followed by:
732//!    - `-hex` followed by hex signature
733//!    - `-file` followed by the path in dedicated [`FOLDER`](constants::FOLDER)
734//!       for file with signature as raw bytes
735//!
736//! - Optional key `-name` followed by path override for export file in
737//!   dedicated [`EXPORT_FOLDER`](constants::EXPORT_FOLDER)
738//!
739//! ### `sign` command
740//!
741//! `$ cargo run make <keys> <arguments>`
742//!
743//! Keys to be used in command line:
744//!
745//! - Key `--goal` followed by the type to generate
746//!    - `qr` will generate only a png QR code
747//!    - `text` will generate only text file with hex-encoded update.
748//!    - default, i.e. if goal is not provided, both QR code and text file are generated.
749//!
750//! - Key `-sufficient` followed by:
751//!    - `-hex` followed by hexadecimal string with contents of Vault-produced
752//!       `SufficientCrypto` QR code
753//!    - `-file` followed by file path in dedicated
754//!       [`FOLDER`](constants::FOLDER) for raw bytes file with contents of
755//!       Vault-produced `SufficientCrypto` QR code
756//!
757//! - Key `-msg` followed by message type:
758//!    - `load-types`
759//!    - `load-metadata`
760//!    - `add-specs`
761//!
762//! - Key `--payload` followed by file path in dedicated
763//!   [`FOLDER`](constants::FOLDER) containing already generated payload as
764//!   raw bytes
765//!
766//! - Optional key `-name` followed by path override for export file in
767//!   dedicated [`EXPORT_FOLDER`](constants::EXPORT_FOLDER)
768//!
769//! Generating `SufficientCrypto` in Vault is suggested mainly for update
770//! distribution purposes. A dedicated (i.e. used only for updates signing),
771//! kept physically safe Vault is strongly suggested, with a dedicated key
772//! for updates signing. As the Vault can accept only payloads with
773//! verifier not weaker than the one used before, and the whole purpose of
774//! the process is to generate a signature for payload, it is expected that
775//! this isolated Vault will receive unsigned or weakly signed updates,
776//! thoroughly check them and export `SufficientCrypto`, so that a signed
777//! update could be made for other, routinely used Vault devices.
778//!
779//! ### Examples: generate `load_metadata` QR code for westend metadata version 9200.
780//!
781//! Update payload `sign_me_load_metadata_westendV9200` is already in dedicated
782//! [`FOLDER`](constants::FOLDER).
783//!
784//! #### `make` for external signature
785//!
786//! `$ cargo run make --goal qr --crypto <encryption> --msg load-metadata
787//! --verifier-hex <public key> --payload sign_me_load_metadata_westendV9200
788//! --signature-hex <signature>`
789//!
790//! Here `<signature>` is hexadecimal signature generated for the contents of
791//! the payload file for `<public_key>` using `<encryption>` algorithm.
792//!
793//! Output file is `load_metadata_westendV9200` in
794//! [`EXPORT_FOLDER`](constants::EXPORT_FOLDER).
795//!
796//! Example:
797//!
798//! `$ cargo run make --goal qr --crypto sr25519 --msg load-metadata --verifier-hex
799//! 46ebddef8cd9bb167dc30878d7113b7e168e6f0646beffd77d69d39bad76b47a --payload
800//! sign_me_load_metadata_westendV9200 --signature-hex
801//! 125717599cd057bfe6db7b111274cbda796d2543467400110552fa1c62dc087a7acefb53b68716f1e34f8af6bf13ab45d70d50655fd39483c64f3f057418748a`
802//!
803//! #### `make` for test verifier Alice
804//!
805//! Payloads signed by Alice are used for testing in Vault. The signature
806//! in this case is generated automatically and is not supplied in command
807//! line.
808//!
809//! `$ cargo run make --goal qr --crypto <encryption> --msg load-metadata
810//! --verifier Alice --payload sign_me_load_metadata_westendV9200`.
811//!
812//! Output file is `load_metadata_westendV9200_Alice-<encryption>` in
813//! [`EXPORT_FOLDER`](constants::EXPORT_FOLDER).
814//!
815//! Example:
816//!
817//! `$ cargo run make --goal qr --crypto sr25519 --msg load-metadata --verifier Alice
818//! --payload sign_me_load_metadata_westendV9200`
819//!
820//! #### `make` with no signature
821//!
822//! `$ cargo run make --goal qr --crypto none --msg load-metadata --payload
823//! sign_me_load_metadata_westendV9200`
824//!
825//! Output file is `load_metadata_westendV9200_unverified` in
826//! [`EXPORT_FOLDER`](constants::EXPORT_FOLDER).
827//!
828//! Example:
829//!
830//! `$ cargo run make --goal qr --crypto none --msg load-metadata --payload
831//! sign_me_load_metadata_westendV9200`
832//!
833//! #### `sign`
834//!
835//! Here `<hex_sufficient>` is hex-encoded data from
836//! [`SufficientCrypto`](definitions::crypto::SufficientCrypto) QR code produced
837//! by the Vault.
838//!
839//! `$ cargo run sign --goal qr --sufficient-hex <hex_sufficient> --msg
840//! load-metadata --payload sign_me_load_metadata_westendV9200`
841//!
842//! Output file is `load_metadata_westendV9200` in
843//! [`EXPORT_FOLDER`](constants::EXPORT_FOLDER).
844//!
845//! Example:
846//!
847//! `$ cargo run sign --goal qr --sufficient-hex
848//! 0146ebddef8cd9bb167dc30878d7113b7e168e6f0646beffd77d69d39bad76b47aceef7c58b5f952b6233b8aba5beb6f0000c8ca7f7cc16b7ada7cd45026fc3f3ec2289dd90dab0dfac38dfe3be843231443ddd30a3f3bbabb5cefcd2bbcef908c
849//! --msg load-metadata --payload sign_me_load_metadata_westendV9200`
850//!
851//! ## Remove a single metadata entry from the `METATREE`
852//!
853//! `$ cargo run remove --name <network_name> --version <metadata_version>`
854//!
855//! Removes only the specified entry from the [`METATREE`](constants::METATREE).
856//!
857//! The entry in [`META_HISTORY`](constants::META_HISTORY) remains. Should the
858//! same metadata version be retrieved afterwards, the `META_HISTORY` entry will
859//! be updated to a block hash from more recent fetch, as the metadata from
860//! old block saved in the database would not be necessarily the same as the one
861//! being recorded in the database now.
862//!
863//! ## Remove all data associated with a network
864//!
865//! `$ cargo run remove --title <address_book_title>`
866//!
867//! This will remove:
868//! - address book entry
869//!   [`AddressBookEntry`](definitions::metadata::AddressBookEntry) from
870//!   [`ADDRESS_BOOK`](constants::ADDRESS_BOOK) tree
871//! - network specs
872//!   [`NetworkSpecs`](definitions::network_specs::NetworkSpecs)
873//!   from [`SPECSTREEPREP`](constants::SPECSTREEPREP) tree
874//! - all associated metadata entries from [`METATREE`](constants::METATREE)
875//!   if there are no other address book entries this metadata is associated
876//!   with
877//! - all associated meta block history entries from
878//!   [`META_HISTORY`](constants::META_HISTORY) if there are no other address book
879//!   entries this block history entries are associated with
880//!
881//! ## Restore hot database to default state
882//!
883//! `$ cargo run restore-defaults`
884//!
885//! Removes old hot database and generates new one with default values at
886//! default path [`HOT_DB_NAME`](constants::HOT_DB_NAME).
887//!
888//! By default, hot database contains:
889//!
890//! - [`ADDRESS_BOOK`](constants::ADDRESS_BOOK) entries for default networks
891//! - [`SPECSTREEPREP`](constants::SPECSTREEPREP) entries for default networks
892//! - types information in [`SETTREE`](constants::SETTREE)
893//! - **no** metadata entries in [`METATREE`](constants::METATREE)
894//! - **no** meta block history entries in
895//!   [`META_HISTORY`](constants::META_HISTORY)
896//!
897//! Default networks are Polkadot, Kusama, and Westend.
898//!
899//! ## Generate default cold release database
900//!
901//! `$ cargo run make-cold-release <optional path>`
902//!
903//! Removes old cold release database and generates new one with default values
904//! (unitiniated) at user-provided path or, if no valid path is given, at
905//! default path [`COLD_DB_NAME_RELEASE`](constants::COLD_DB_NAME_RELEASE).
906//!
907//! By default, the uninitiated cold release database contains:
908//!
909//! - [`SPECSTREE`](constants::SPECSTREE) entries for default networks
910//! - [`VERIFIERS`](constants::VERIFIERS) entries for default networks, with
911//!   verifiers set to the general one
912//! - two latest metadata versions for default networks in
913//!   [`METATREE`](constants::METATREE)
914//! - default types information and clean danger status in
915//!   [`SETTREE`](constants::SETTREE)
916//!
917//! Note that the general verifier is not specified and history is not
918//! started. This will be done only in Vault itself. Before initialization,
919//! the cold release database could not be used by Vault.
920//!
921//! ## Transfer metadata from hot database to cold release database
922//!
923//! `$ cargo run transfer_meta_to_cold_release <optional path>`
924//!
925//! Metadata from hot database is transferred to cold release database at
926//! user-provided path or, if no valid path is given, at default path
927//! [`COLD_DB_NAME_RELEASE`](constants::COLD_DB_NAME_RELEASE).
928//!
929//! Metadata is transferred only for the networks that are known to the cold
930//! database, i.e. the ones having
931//! [`OrderedNetworkSpecs`](definitions::network_specs::OrderedNetworkSpecs) entry in
932//! [`SPECSTREE`](constants::SPECSTREE).
933
934//! ## Make derivations import QR and/or hexadecimal string file
935//!
936//! `$ cargo run derivations --goal <GOAL> --title <TITLE> --derivations <DERIVATIONS>`
937//!
938//! Keys to be used in command line:
939//!
940//! - `<GOAL>`: `qr` will generate only apng QR code, `text`
941//!   will generate only text file with hex-encoded update. By default, i.e. if
942//!   content key is not provided, both QR code and text file are generated.
943//!   `<optional_target_key>` is expected immediately after `derivations` command,
944//!   if at all; keys to follow could go in any order, but with argument
945//!   immediately following the key.
946//!
947//! - Key `--derivations` followed by file path in `/generate_message/` folder.
948//!   File with derivations contains valid derivations, each on its own line. Only
949//!   suitable derivations will be processed. Processed derivations are also
950//!   printed for user to check.
951//!
952//! - Key `--title` followed by network address book title, to indicate to
953//!   which network the derivations belong.
954//!
955//! Output file is in `/generate_message/` folder, file name would be
956//! `derivations-<address_book_title>`.
957//!
958//! ## Prepare payload for `load_metadata` update from `.wasm` file
959//!
960//! `$ cargo run unwasm [OPTIONS] --filename <FILENAME>`
961//!
962//! This command extracts metadata from `.wasm` file and uses this metadata to
963//! produce `load_metadata` update payload. Only networks with network specs
964//! entries in the hot database could be processed with `unwasm` command, since
965//! the `load_metadata` update payload in addition to metadata requires also
966//! network genesis hash. `unwasm` command could be used to generate update QR
967//! codes before the metadata becomes accessible from the node.
968//!
969//! Network name found in the metadata is used to find
970//! [`NetworkSpecs`](definitions::network_specs::NetworkSpecs) for
971//! the network. `NetworkSpecs` are used to get genesis hash and to check
972//! base58 prefix, it the network metadata has base58 prefix inside.
973//!
974//! A raw bytes update payload file is generated in dedicated
975//! [`FOLDER`](constants::FOLDER) to (optionally) be signed and later be
976//! transformed into `load_metadata` update QR. Update payload file name is
977//! `sign_me_load_metadata_<network_name>V<version>`.
978//!
979//! By default, metadata extracted from `.wasm` file is added to the database.
980//! Optional `-d` key could be used is database should **not** be updated.
981//! If the metadata gets entered in the database (i.e. no `-d` key used),
982//! [`META_HISTORY`](constants::META_HISTORY) gets no entry. Block hash will be
983//! added if the same metadata is later fetched from a node.
984//!
985//! ## Make metadata file for `defaults` release metadata set
986//!
987//! `$ cargo run  meta-default-file --name <NETWORK NAME> --version <NETWORK VERSION>`
988//!
989//! Produces file with hex-encoded network metadata from the hot database
990//! [`METATREE`](constants::METATREE) entry.
991//!
992//! Output file named `<network_name><metadata_version>` is generated in
993//! dedicated [`EXPORT_FOLDER`](constants::EXPORT_FOLDER). It contains
994//! hexadecimal network metadata.
995//!
996//! ### Example
997//!
998//! `$ cargo run meta-default-file --name westend --version 9230`
999//!
1000//! ## Make file with hexadecimal network metadata fetched for specific block hash from provided address
1001//!
1002//! `$ cargo run meta-at-block --url <RPC URL> --block-hash <BLOCK HASH>`
1003//!
1004//! Output file named `<network_name><metadata_version>_<block_hash>` is
1005//! generated in dedicated [`EXPORT_FOLDER`](constants::EXPORT_FOLDER).
1006//! It contains hexadecimal network metadata.
1007//!
1008//! This command does not address or update the hot database.
1009//!
1010//! ### Example
1011//!
1012//! `$ cargo run meta-at-block --url wss://westend-rpc.polkadot.io --block
1013//! 780812df50c4006d1865742269fe4ca339c097e61d6279cce91ebc58f5aebada`

1046/// Process incoming command as interpreted by parser.
1047pub fn full_run(command: Command) -> Result<()> {
1048    match command {
1049        Command::Show { s: show, db_path } => {
1050            let database = sled::open(db_path)?;
1051            match show {
1052                Show::Metadata => show_metadata(&database),
1053                Show::Networks => show_networks(&database),
1054                Show::Specs { s: title } => show_specs(&database, title),
1055                Show::CheckFile { s: path } => check_file(&database, path),
1056                Show::BlockHistory => show_block_history(&database),
1057            }
1058        }
1059        Command::Specs { s: instruction } => gen_add_specs(instruction),
1060        Command::Load(instruction) => gen_load_meta(instruction),
1061        Command::Types { db_path, files_dir } => {
1062            let database = sled::open(db_path)?;
1063            Ok(prep_types(&database)?.write(files_dir.join("sign_me_load_types"))?)
1064        }
1065        Command::Sign(make) | Command::Make(make) => make_message(make),
1066        Command::Remove { r: info, db_path } => {
1067            let database = sled::open(db_path)?;
1068            remove_info(&database, info)
1069        }
1070        Command::RestoreDefaults { db_path } => {
1071            let db = sled::open(db_path)?;
1072
1073            Ok(default_hot(Some(&db))?)
1074        }
1075        Command::MakeColdRelease { path } => {
1076            let db = path.and_then(|path| sled::open(path).ok());
1077
1078            Ok(default_cold_release(db.as_ref())?)
1079        }
1080        Command::TransferMetaToColdRelease { cold_db, hot_db } => {
1081            let hot_db = sled::open(hot_db)?;
1082            let cold_db = sled::open(cold_db)?;
1083            Ok(transfer_metadata_to_cold(&hot_db, &cold_db)?)
1084        }
1085        Command::Unwasm {
1086            filename,
1087            update_db,
1088            db_path,
1089            files_dir,
1090        } => {
1091            let database = sled::open(db_path)?;
1092            unwasm(&database, &filename, update_db, files_dir)
1093        }
1094        Command::MetaDefaultFile {
1095            name,
1096            version,
1097            db_path,
1098            export_dir,
1099        } => {
1100            let database = sled::open(db_path)?;
1101            meta_default_file(&database, &name, version, export_dir)
1102        }
1103        Command::MetaAtBlock {
1104            url,
1105            block_hash,
1106            export_dir,
1107        } => debug_meta_at_block(&url, &block_hash, export_dir),
1108        Command::EncodeToQr {
1109            path,
1110            hex,
1111            chunk_size,
1112            dst_file,
1113        } => {
1114            let data = if let Some(hex) = hex {
1115                hex::decode(hex).unwrap()
1116            } else if let Some(path) = path {
1117                std::fs::read(path).unwrap()
1118            } else {
1119                panic!("path or hex data required");
1120            };
1121
1122            generate_qr_code(&data, chunk_size, FPS_DEN, dst_file)
1123        }
1124        Command::KeyInfoExportToQr {
1125            dst_file,
1126            chunk_size,
1127            fps,
1128            keys_num,
1129        } => generate_key_info_export_to_qr(dst_file, chunk_size, fps, keys_num),
1130        Command::BulkTransactionTestPayload {
1131            dst_file,
1132            tx_count,
1133            chunk_size,
1134            from,
1135            output_format,
1136        } => generate_bulk_transaction_qr(dst_file, tx_count, chunk_size, from, output_format),
1137    }
1138}
